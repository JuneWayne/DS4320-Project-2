## Executive Summary

This repository contains the full pipeline for a binary text classification project that tries to tell apart human-written text from AI-generated text. The dataset comes from the RAID benchmark (Dugan et al., ACL 2024), sampled down to 16,000 documents across 8 content domains and 6 AI model sources, and stored in MongoDB Atlas. The pipeline combines TF-IDF text features with handcrafted linguistic statistics to train a Logistic Regression classifier, which is then tested on a held-out model family that was never seen during training. The repository includes the data loading script, the analysis pipeline notebook, metadata documentation, and a press release.

## Problem Definition

**General Problem:** How to detect fake or disingenuous content that is not faithfully or truthfully created by students in exams or community creators on content sharing platforms?

**Specific Problem:** Given a piece of text, how should we accurately assess and classify whether it was written by a human or generated by a large language model, using linguistic and stylometric features extracted from the text itself?

### Motivation

LLMs like OpenAI's GPT series, Claude, and Gemini can now produce text, sentences, paragraphs, or phrases that are almost indistinguishable from human written text. This created an unprecedented challenge in detecting academic honesty, curbing the spread of misinformation and misguided information on social media, fake reviews, and other systematically generated propaganda that misguides individuals to consume harmful information from social media platforms. While there are existing detection tools like GPTZero and Turnitin that allegedly can detect AI generated content, their platforms are often considered to be outdated and can be easily fooled by more advanced LLMs that are being developed by the day, or are simply unreliable and cannot effectively distinguish human text from AI generated text. Given an unprecedented challenge in trustworthiness of online content, a more advanced classifier and AI text detector must be developed to accommodate the needs of filtering fake or disingenuous content online.

### Rationale for Refinement

The refinement is inspired by how most of the misinformation spread online is generated from so called "bots", or automated social media accounts that execute scripts every day to create non-human user generated texts. This phenomenon is especially aided and facilitated by the advent of AI. At the same time, having a general problem of "identifying fake content" is just too broad. It could encompass images, audio, video, and text all at once. Given the limitations of the data available and how it might be harder to process multi-modal data in JSON databases, it would be more practical and realistic to start with detecting AI generated textual data. At the same time, this is also one of the most challenging types of data to detect, given how easy it is to mimic human generated text. Hence, I've decided to narrow down the problem to detecting AI generated text using a range of features and style related data extracted from both human generated text and AI generated text.

### Domain Overview

This project sits at the intersection of natural language processing and machine learning security. As LLMs become more capable and widely used, the ability to reliably detect AI-generated text becomes increasingly important for maintaining trust in written communication. The field is moving fast, and detection methods that work well on one generation of models often fail on newer ones. This makes cross-model generalization one of the central challenges in this space.

## Data Creation

The dataset used in this project is derived from the RAID (Robust AI-Generated text Detection) benchmark dataset introduced by Dugan et al. in their 2024 paper about finding robust ways to detect AI generated text. The raw data was downloaded directly from the RAID paper's open sourced dataset `train.csv`, which contains over 5 million text generations produced by 11 large language models across multiple domains and decoding strategies, alongside human-written source texts from the same domains. Due to memory constraints, a stratified sample of 2,000 documents per domain was collected from each of the 8 domains. This resulted in a 16,000 document database across the 8 domains that I believe will be sufficient for training a model and having it learn the patterns of AI and human generated text. The ground truth labels were derived from the model field — human written text were assigned 0 and all other texts were assigned label 1. The dataset is loaded onto MongoDB into three major collections: `texts`, `models`, and `domains`.

### Rationale for Critical Decisions

The decision to sample 2,000 documents per domain was driven by memory constraints on the local machine used to load the downloaded data. Loading a full 5 million row CSV continuously crashed the kernel, so chunked loading per sampled domain was the only viable approach. Although this may introduce uncertainty because the sample may not be fully representative of the full domain distribution, this still needs to be improved in future work by conducting bootstrap resampling techniques to ensure full randomization.

The decision to derive labels from the model field rather than using an explicit label column was necessary because the RAID train set does not provide direct ground truth labels, so a one hot encoding was created for each type of text instead.

For the train/test split, a model-based split was used rather than a random row split. Specifically, llama-chat was held out entirely as the test model since it comes from Meta and uses a completely different architecture from the other models in the dataset. This was a deliberate choice to test whether the classifier learned general properties of AI writing rather than just memorizing vocabulary patterns from models it had already seen. Human documents were split 80/20 across train and test to ensure both classes were represented in the test set.

The decision to drop decoding strategy and repetition penalty from the feature set came after noticing that these columns directly leaked the label. Human rows always have these fields set to null, while AI rows always have them filled in, meaning the model could classify almost perfectly from those two columns alone without learning anything about the text itself. Removing them forces the model to rely on TF-IDF and genuine linguistic features instead.

The TF-IDF vectorizer was fit only on the training set and applied via transform-only on the test set. This can prevent any vocabulary from the test documents leaking into the model during training, which would otherwise inflate performance metrics.

Finally, rather than manually tuning the class weight and regularization parameter C, a grid search was run over human class weights of 1, 3, 5, 7, 10, 15, and 20, and C values of 0.01, 0.05, 0.1, 0.5, and 1.0, evaluating each combination on the validation set using PR-AUC. PR-AUC was chosen over accuracy or ROC-AUC because it is more informative for imbalanced datasets, focusing specifically on how well the model identifies the minority class. The best combination from the search was class weight of 20 for human and C of 1.0.

### Bias Identification

Bias could have been introduced at the data collection stage by having an unbalanced class of AI generated and human written text data. About 90% of the documents in the text collection are labeled as AI, while only about 10% are actually human written. This may introduce overrepresentation of one class which might lead to high false positive rates due to AI generated text being statistically more prevalent in the training set. Secondly, human written texts are drawn from only a few specific academic and online resources such as arXiv abstracts, Reddit, and Wikipedia, meaning they only represent particular writing styles and demographics rather than general human writing style. Lastly, models are constantly improving. The use of early stage models like gpt2 and llama, which are already obsolete and not used in production, could bias the model to believe that AI generated text is easily distinguishable from human text, to which more advanced models nowadays will definitely prove otherwise with advanced reasoning capabilities.

### Bias Mitigation

The class imbalance between human and AI generated text was directly addressed during modeling in two ways. First, a stratified split was used when dividing the human documents across train and test sets, ensuring that the 9/91 human to AI ratio was preserved consistently across all three splits rather than having one split end up with disproportionately fewer human samples. Second, a class weight of 20 for the human class was applied in the Logistic Regression, which was selected through a grid search over multiple weight and regularization combinations evaluated on the validation set using PR-AUC. This tells the model to penalize misclassifying a human sample 20 times more heavily than misclassifying an AI sample, pushing it to take the minority class more seriously during training.

The domain specific bias in human writing sources can be quantified by evaluating model performance in each domain, which allows identification of domains where the classifier underperforms and domains where it performs well. This way we could see which domains or writing styles still need more data collection in order to have the model successfully distinguish AI generated text from human text. To address the model obsolescence bias, the dataset must simply be enriched with a more abundant level of AI generated texts from current SOTA models like Claude Opus, GPT-5, or DeepSeek R-1, to allow the model to learn patterns from AI generated text that are from advanced reasoning equipped capability models.

## Metadata

### Implicit Schema Guidelines

The `texts` collection is the primary collection of the database. It contains the actual text blob, what was used to generate it, which domain the text belongs to, and what are the statistical features of the texts. The `texts` collection conforms to the following structure:

- Every document must contain fields `raid_id`, `model_id`, `domain_id`, `text`, and `label`. The `raid_id` field must be a UUID string that uniquely identifies the source generation.
- The `model_id` and `domain_id` fields are foreign keys referencing documents in the `models` and `domains` collections. They must have an existing corresponding entry in those collections.
- The `label` field must be an integer of 0 for human written text and 1 for AI generated text.
- The linguistic features `word_count`, `avg_sentence_length`, `unique_word_ratio`, `avg_word_length`, and `punctuation_density` must be represented as floating point numbers for all documents.
- The fields `repetition_penalty` and `decoding` are only applicable to AI generated texts, not human written texts.
- Both the `models` and `domains` collections must contain one document per unique model or domain with no duplicates.
